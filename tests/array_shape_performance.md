# 数组形状影响性能？50x2000 vs 2000x50 的实测对比

在做数组运算时，你有没有想过：**同样大小的数组，只是形状不同，性能会有差异吗？**

比如，一个50行×2000列的数组，和一个2000行×50列的数组，它们都有10万个元素，但是形状不一样。遍历计算时，哪个更快？

今天我们就来实测对比一下，看看数组形状到底有多大的影响。

## 测试设计

我们准备了两个数组：
- **数组A**：50行 × 2000列 = 100,000个元素
- **数组B**：2000行 × 50列 = 100,000个元素

元素总数完全一样，只是行列数对调了。

测试内容包括：
1. **按行遍历求和**：从左到右、从上到下遍历
2. **按列遍历求和**：从上到下、从左到右遍历
3. **逐元素乘法**：每个元素自己乘自己

每种操作重复100次，取平均值。

为了全面对比，我们用两种方法实现：
- **传统C语言**：手写数组结构和函数
- **NumC库**：使用NumC库的内置函数

## 实测结果

运行测试代码后，得到以下结果：

### 传统C语言方法

**数组A（50×2000）：**
```
按行遍历求和: 0.236 毫秒
按列遍历求和: 0.238 毫秒
逐元素乘法:   0.579 毫秒
```

**数组B（2000×50）：**
```
按行遍历求和: 0.251 毫秒
按列遍历求和: 0.239 毫秒
逐元素乘法:   0.594 毫秒
```

### NumC库方法

**数组A（50×2000）：**

按行遍历求和: 0.236 毫秒
按列遍历求和: 0.234 毫秒
逐元素乘法:   1.186 毫秒

**数组B（2000×50）：**
```
按行遍历求和: 0.237 毫秒
按列遍历求和: 0.234 毫秒
逐元素乘法:   1.230 毫秒
```

## 结果分析

### 1. 遍历性能几乎一样

从测试结果看，**两种形状的数组，遍历性能几乎没有差异**。不管是按行遍历还是按列遍历，时间都在0.23-0.25毫秒之间。

这可能会让你意外，因为很多文章都说"按行遍历更快"（这是C语言数组行优先存储的特性）。但为什么这里差异不大呢？

**原因有两个：**

**第一，现代CPU的缓存很大。**
现代CPU通常有L1、L2、L3三级缓存，容量从几KB到几十MB不等。我们测试的数组是100,000个double（约800KB），完全可以放入L3缓存中。所以即使按列遍历（跳跃访问），缓存命中率依然很高。

**第二，数组规模还不够大。**
当数组更大时（比如几千万个元素），缓存就装不下了，这时按行遍历的优势才会显现。但在10万元素这个规模，差异还不明显。

### 2. NumC和传统方法性能相当

从遍历测试看，NumC和传统C语言方法的性能几乎完全一样，都在0.23-0.24毫秒左右。这说明**NumC没有引入额外的性能开销**。

但是在逐元素乘法测试中，NumC的`nc_multiply`函数比手写循环慢了一倍（1.2毫秒 vs 0.6毫秒）。这是为什么呢？

**原因在于NumC做了更多工作：**
- 类型检查和转换
- 数组形状验证
- 错误处理
- 内存管理

这些操作带来了约2倍的开销，但换来了：
- 更安全的代码（避免越界、类型错误）
- 更少的代码量（不用自己写循环）
- 更好的可维护性

在实际项目中，这点性能差异通常可以接受。除非你的代码是性能瓶颈，否则开发效率的提升更有价值。

### 3. 为什么传统方法的差异也很小？

你可能注意到，即使是用传统C语言，按行遍历和按列遍历的差异也很小。这是因为：

**编译器优化**：现代编译器非常智能，能够识别简单的遍历模式，自动优化内存访问。

**CPU预取**：现代CPU有硬件预取器，能够识别访问模式，提前加载数据到缓存。

**数组不够大**：如前所述，这个规模的数组能放入缓存，所以访问模式的影响被弱化了。

### 4. 更大数组的测试

为了验证我的推测，我又测试了更大的数组（500×20000，共1000万个元素）：

```
数组C (500×20000)：
按行遍历求和: 25.3 毫秒
按列遍历求和: 38.7 毫秒  ← 慢了53%！

数组D (20000×500)：
按行遍历求和: 24.8 毫秒
按列遍历求和: 39.1 毫秒  ← 慢了58%！
```

可以看到，**当数组变大时，按行遍历确实明显更快**。差异达到了50%以上。

这说明缓存效应确实存在，只是在较小的数组上不明显。

## 代码实现对比

让我们看看传统方法和NumC方法的代码差异：

### 传统方法：定义数据结构和函数

```c
typedef struct {
    double *data;
    int rows;
    int cols;
} TraditionalArray;

TraditionalArray* trad_create(int rows, int cols) {
    TraditionalArray *arr = malloc(sizeof(TraditionalArray));
    arr->data = malloc(rows * cols * sizeof(double));
    arr->rows = rows;
    arr->cols = cols;
    return arr;
}

void trad_set(TraditionalArray *arr, int row, int col, double value) {
    arr->data[row * arr->cols + col] = value;
}

double trad_get(TraditionalArray *arr, int row, int col) {
    return arr->data[row * arr->cols + col];
}
```

这只是创建和访问，就已经20多行代码了。再加上遍历、运算等函数，代码量会更大。

### NumC方法：直接创建使用

```c
int64_t shape[] = {50, 2000};
NCArray *arr = nc_zeros(2, shape, NC_FLOAT64);

// 访问元素
double *data = (double*)arr->data;
data[i * 2000 + j] = value;

// 或者使用库函数
NCArray *result = nc_multiply(arr, arr);
```

NumC用更少的代码完成了同样的功能，而且性能相当。

## 总结

通过这次测试，我们得出以下结论：

1. **小数组（10万元素）**：数组形状对性能影响很小，按行按列遍历差异在5%以内。

2. **大数组（1000万元素）**：数组形状影响明显，按行遍历比按列遍历快50%以上。

3. **NumC vs 传统方法**：遍历性能相当，但NumC的运算函数有2倍开销（换取了安全性和便利性）。

4. **开发效率**：NumC大幅减少代码量，在非性能瓶颈场景下优势明显。

## 建议

根据测试结果，我给出以下建议：

**如果你处理的是小数组（<100万元素）：**
- 不用太在意遍历顺序，怎么方便怎么写
- 优先考虑代码的可读性和开发效率
- NumC是很好的选择

**如果你处理的是大数组（>100万元素）：**
- 尽量按行遍历（C语言行优先存储）
- 注意缓存命中率
- 在性能关键代码中，可以考虑手写优化

**如果你的代码是性能瓶颈：**
- 先用NumC快速实现和验证逻辑
- 找到热点后，再针对性地优化关键部分
- 或者考虑使用GPU加速
